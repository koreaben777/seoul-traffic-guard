공식 기준 요약:

- 창의성: 새로운 아이디어로 문제를 해결하고 파급력을 가지는가
- 편의성: 적합한 UI/UX를 통해 사용자의 일상에 실질적인 가치를 주는가
- 안정성: 안정적으로 구동되며 정확한 데이터를 제공하고 보안상 문제가 없는가

## 현재 상태 요약

`서울 교통 이슈 알리미`는 사용자가 카카오톡 대화로 생활권 또는 대중교통 경로를 등록하고, 서울 실시간 교통 이슈를 확인하거나 본인 카카오톡으로 알림을 받을 수 있는 Remote MCP 서버다.

현재 구현:

- 주소/장소 반경 기반 알림 지역 등록
- ODsay 대중교통 후보 경로 조회 및 선택 등록
- 서울 교통 이슈 조회
- KakaoTalk 나에게 보내기 OAuth 발송
- 예약 알림 등록/조회/삭제
- 공개 tool 10개

## 강점

- 카카오톡 안에서 바로 쓸 수 있는 생활 밀착형 문제다.
- MCP tool 수가 10개로 정리되어 PlayMCP 권장 범위에 맞다.
- 사용자가 위치를 직접 확정하고, 대중교통 경로도 후보 중 선택하게 하므로 오등록 위험이 낮다.
- 실제 알림 발송은 OAuth와 dry-run 경로를 분리해 심사 중 오발송 리스크를 줄였다.
- 서울 Open API, Kakao Local API, ODsay를 조합해 단일 공개 데이터보다 넓은 사용 흐름을 만든다.

## 리스크

- 배포 환경에서 외부 API 허용 IP가 달라지면 ODsay가 다시 막힐 수 있다. - ODsay에서 IP 추가는 쉽게 가능함
- 예약 알림은 서버 프로세스가 살아 있는 동안 동작하는 구조라, 장기 운영 안정성은 PlayMCP in KC 런타임 특성에 의존한다.
- KakaoTalk 실제 발송은 사용자 OAuth 상태에 민감하다. 심사자가 OAuth를 끝내지 않으면 발송 tool은 실패 메시지를 보게 된다.
- 사용자별 데이터 분리는 PlayMCP가 신뢰 가능한 user id header를 주입한다는 전제에 의존한다.
- 실시간 교통 이슈가 없는 시간/지역에서는 서비스 가치가 약하게 보일 수 있다.
- 현재는 서울시 교통 이슈 중심이라 전국 확장성은 낮다.

## 등록 전 보완 우선순위

1. 배포 endpoint에서 ODsay 후보 조회와 선택 등록을 다시 확인한다.
2. 배포 endpoint의 Kakao OAuth callback과 실제 나에게 보내기 발송을 다시 확인한다.
3. PlayMCP user id header 이름을 확인하고 `PLAYMCP_USER_ID_HEADERS`, `REQUIRE_USER_ID_HEADER=true`를 배포 환경에 설정한다.
4. PlayMCP AI 채팅에서 대표 시연 문장 5개를 통과시킨다.
5. 심사 설명에는 "서울 한정", "나에게 보내기 한정", "대중교통 후보 선택형"을 명확히 적는다.
6. 실시간 이슈가 없을 때도 가치가 보이도록 예시 문장에는 `서울시청`, `을지로 19`, `소공로`처럼 현재 이슈가 자주 잡히는 중심지를 사용한다.

## 추천 시연 문장

```text
출근길을 서울시청 반경 1km로 등록해줘.
출근길 주변 교통 이슈를 확인해줘.
이 내용을 실제 발송 말고 발송 전 확인만 해줘.
이촌로 174에서 을지로 19까지 대중교통 경로 후보를 보여줘.
2번 경로를 500번 출근길로 등록해줘.
```

## 결론

예선 제출 가능한 수준이다. 다만 심사 전 마지막 관문은 로컬이 아니라 배포 endpoint 기준의 ODsay, OAuth, 실제 발송 확인이다. 이 세 가지가 통과하면 안정성 점수의 가장 큰 리스크가 줄어든다.
